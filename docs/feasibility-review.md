# 災害時避難シミュレーションWebアプリ — 技術実現性レビュー

> システムアーキテクト視点での厳しめ評価。実現性・コスト・データ入手性・開発難易度を踏まえる。

## 総評（結論）

**「地図＋ハザードマップ重ね合わせ＋避難所検索」までは確実に作れる（9/10）。しかし提案の核である「危険区域を避けた避難ルート提案」は、無料・個人開発・1か月という制約下では"それっぽく動くデモ"止まりになり、厳密にやろうとすると急に難易度が跳ね上がる。** ここを正直に切り分けられるかがプロジェクトの成否。

| 現在地取得 | 10 | Geolocation API、標準機能 |
| 地図表示 | 10 | MapLibre / Leaflet で確実 |
| 災害種別選択 | 10 | ただの状態管理 |
| 避難所検索 | 9 | 国土数値情報の避難所データが使える |

「全部できます」ではなく「ルート探索だけが別次元の難しさ」という認識を最初に持つべき。

## 2. Googleマップとの差別化

方向性は正しいが詰めが甘いと「ハザードマップを重ねただけのGoogleマップ」に見える。

- Googleマップは「危険を考慮した経路探索」を**やらない**ので、そこは明確な差別化。
- ただし国交省「重ねるハザードマップ」が既に存在し、**ハザード表示＋避難所表示は公式が無料提供済み**。単なる重ね合わせでは「国のサイトで十分」と突っ込まれる。
- **差別化の生命線は「危険を避けるルート＋推奨理由の明示」の一点**。ここが動けば勝ち。

---

## 3. 最大の技術的課題

**「ハザードのポリゴンデータを、歩行者ルート探索エンジンの回避条件として食わせること」** に集約される。

1. **ハザードデータの形式問題** — 公式配信の多くは画像タイル（ラスタ）。ルート回避に必要なのはベクタ（ポリゴン）。タイルは「表示」できても「この座標は浸水域か？」の判定に直接使えない。→ 国土数値情報の GML/Shapefile をポリゴンとして取得し直す必要があり、データ量が重い・全国対応は非現実的（デモは1自治体に絞る）。
2. **津波・土砂は「回避」では不十分** — 津波は"高台へ逃げる"のが正解で、標高を目的関数に組み込む必要がある。土砂は谷筋・斜面を避ける。**1つの回避ロジックで4災害対応は破綻する。**
3. **経路探索エンジンの回避機能の制約** — OpenRouteService の `avoid_polygons` はポリゴン数・面積・APIの制限があり、市街地全域の浸水ポリゴンを渡すと413エラーや無視される。
4. **災害時に通信は生きているのか** — 本番でアプリは動くのか。オフライン対応しないなら「平常時の学習ツール」と位置づけを正直に宣言すべき。

## 4. 無料で使えるAPI・オープンデータ・ライブラリ

### 地図・ベースレイヤ
- **地理院タイル（国土地理院）** — 標準/淡色/写真。利用申請不要で商用可。日本の防災系なら第一選択。
- **MapLibre GL JS**（本アプリ採用）or Leaflet — 無料OSS。
- **OpenStreetMap** — 経路探索の道路網ソース。

### ハザード（表示用タイル）
- **ハザードマップポータル／重ねるハザードマップのタイル** — 洪水浸水想定・津波浸水想定・土砂災害警戒区域・高潮などをXYZタイルで配信。**貼るだけ。**

### ハザード（判定・回避用ポリゴン）
- **国土数値情報（国土交通省）** — 浸水想定区域、土砂災害警戒区域、津波浸水想定を GML/Shapefile でDL可 → GeoJSON化。
- **指定緊急避難場所・指定避難所データ** — 国土数値情報 or 各自治体オープンデータ（災害種別対応フラグ付き点データ）。

### 標高（津波・高台判定の要）
- **国土地理院 標高API / 標高タイル(DEM)** — 緯度経度から標高取得。津波避難ロジックの中核。

### 経路探索（無料）
- **OpenRouteService（ORS）** — 無料APIキーあり、歩行者プロファイル対応、`avoid_polygons` あり。本命。レート制限・ポリゴンサイズ制限に注意。
- GraphHopper / Valhalla / OSRM — セルフホスト前提。個人開発1か月なら非推奨。

### バックエンド／DB（バックエンドを採る場合）
- **Spring Boot ＋ PostgreSQL + PostGIS** — 空間クエリ（点in多角形・最近傍）をDBに任せる。SQLだけでは死ぬのでPostGIS必須。ogr2ogr(GDAL)でデータ投入、Java側は JTS / GeoTools。

### ホスティング（無料枠）
- フロント：Cloudflare Pages / GitHub Pages / Netlify
- バックエンド：Render / Railway / Fly.io（Spring Bootはコールドスタート注意）
- DB：Supabase(PostGIS可) / Neon

### 気象・災害情報（発展用）
- **気象庁 防災情報XML / JSONフィード** — 警報・注意報のリアルタイム取得。

## 5. 「危険区域を避けたルート探索」は無料で実現できるか

**「できる、ただし条件付き」。**

- 最短で動かす道：**OpenRouteService の `avoid_polygons`** に対象地域の浸水/土砂ポリゴンを渡す（無料APIキーで動く。現実解）。
- 制約（発表で正直に言うべき）：
  - ポリゴンが大きい/多いと制限で回避が効かない → **1自治体・災害1種類に絞る**のが賢い。
  - 回避エリアで避難所が完全に囲まれると経路なし → フォールバック（「安全ルートなし。垂直避難を推奨」）が要る。
  - 津波の「高台へ」は単純回避で表現できない → **標高スコアで避難所選定＋通常歩行ルート**の二段構えに割り切る。

結論：無料で作れるが、"全災害・全国・厳密"は無理。スコープを絞れば発表レベルで成立する。

## 6. 発表で「おっ」と思われるポイント

1. **推奨理由の言語化**（「浸水想定区域を回避」「標高◯m」「徒歩◯分」を根拠付きで）
2. **災害種別で挙動が変わる**（津波は標高重視、洪水は浸水回避）
3. **標高プロファイル**（津波避難で"登っていく"の可視化）
4. **空間処理の中身**（点in多角形・最近傍）
5. **AI活用の説明**（開発への活用＋避難理由文の生成）

---

## 7. 弱い点・突っ込まれそうな点

- 「災害時に本当に使えるのか？」→ **「平常時の避難計画・学習支援ツール」と位置づけを明言**すれば防御可能。
- データの鮮度と責任 → **免責と「参考情報」明記**は必須。
- 回避ルートの妥当性検証 → 「ハザードマップ上の区域を通らない」以上の保証はできないと正直に。
- スコープの狭さ → 「拡張可能な設計だが実データは◯市に限定」と誠実に。
- Spring Bootの過剰感 → 「PostGISで空間処理を堅くやるため」と技術的必然性を語る。
- リアルタイム性の欠如 → 想定区域であり「シミュレーション」の域を出ない。

---

## 8. レベルアップ機能の提案（効果順）

1. **オフライン対応（PWA）** — タイル・データをキャッシュ。最大の弱点を潰せる。**費用対効果No.1。**
2. **垂直避難の提案** — 水平避難不能時に「近くの◯階建て以上へ」。
3. **複数避難所のスコアリング比較** — 距離・標高・災害対応・収容人数を重み付けし上位3件を理由付きで。
4. **AIによる避難アドバイス生成** — Claude API で状況別アドバイス。
5. **「もし今起きたら」タイムライン** — 津波到達まで◯分 vs 徒歩◯分の間に合う判定。
6. **要配慮者モード** — 歩行速度・段差回避を変える。

---

## 9. 卒業制作・社内発表レベルとしての評価

- **卒業制作・学部レベル：十分通用（上位）。** 実データ連携・空間処理・回避ルートまで到達すれば高評価。
- **社内技術発表：中〜上。** 「APIに投げてるだけ」だと組み合わせ扱い。空間処理・災害別ロジック・オフライン対応など「自作した中身」を前面に出せば上位。
- 正直な評価：アイデアの完成度は高いが、差別化の核（回避ルート）が浅いと"公式の劣化コピー"に見えるリスク。**深さで勝負すべき。**

## 10. システム構成図（無料構成・本アプリ採用）

```
┌────────────────────────────────────────────────────────────┐
│                     ユーザー (ブラウザ / PWA)                       │
│   - Geolocation API で現在地取得                                    │
│   - MapLibre GL JS で地図描画・ハザードタイル重畳                      │
│   - Service Worker でキャッシュ（オフライン対応：発展）                 │
└───────────────┬────────────────────────────────────────────┘
                │ 静的配信 (HTTPS)
                ▼
┌────────────────────────────────────────────────────────────┐
│  フロント配信: GitHub Pages / Cloudflare Pages (無料)               │
│  ※ バックエンド・DBなし。全ロジックはクライアントサイド。               │
└──┬──────────────────┬──────────────────┬───────────────────┘
   │ タイル取得          │ 標高取得           │ 経路探索
   ▼                    ▼                   ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐
│ 地理院タイル    │  │ 地理院 標高API │  │ OpenRouteService      │
│ ・地図(pale)   │  │ getelevation │  │ ・foot-walking        │
│ ハザードタイル  │  │ (津波の高台判定)│  │ ・avoid_polygons(洪水) │
│ ・洪水/津波/土砂│  └──────────────┘  │  無料APIキー           │
└──────────────┘                     └──────────────────────┘
        ▲
        │ 避難所データ(GeoJSON) … リポジトリ同梱 or 国土数値情報を差し替え
        └── data/shelters.sample.geojson

【データフロー例：洪水時のルート提案】
 現在地取得 → 災害種別=洪水を選択 → 洪水ハザードタイルを重畳表示
   → 近傍避難所を距離でランク付け → 推奨避難所へORSで歩行ルート計算
     （洪水ポリゴンがあれば avoid_polygons で回避）
   → 標高・距離・災害対応から推奨理由を生成 → 地図に描画＋理由表示
```

構成上の注意：
- ハザードタイルはフロントから直接貼れば表示は無料・即完了。
- 全国データは重いので避難所データ・回避ポリゴンは対象地域だけに絞る。
- ORSは無料枠にレート制限あり。デモ用途なら十分。
