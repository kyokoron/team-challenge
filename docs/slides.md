# 災害時避難シミュレーター

### ハザードを考慮した、避難の意思決定支援 Web アプリ

<br>

Kyoko Takazawa
デモ: https://kyokoron.github.io/team-challenge/

---

## 背景と課題

- 災害時、多くの人は **「どこへ・どう逃げるか」** を即座に判断できない
- 既存の地図サービス（例：Google マップ）は **最短経路** は出せるが
  **災害の危険（浸水・高台・指定避難場所）を考慮しない**
- 国のハザードマップは情報は正確だが **「あなたが今すべき行動」** までは示さない

> 課題：ハザードを踏まえた **意思決定** をその場で支援する仕組みがない

---

## コンセプト

**「危険を避ける」避難案内 ＋ 平常時に備え、被災時に使う設計**

- 災害の種類（地震／津波／台風・高潮／洪水／土砂）に応じて
  **最適な避難所と避難ルート**を提案
- 単なる最短経路ではなく、**ハザードを考慮した意思決定支援**
- 「ミスが許されない」前提で、**誤案内を生む要素を排除**する安全設計

---

## 主な機能

- 現在地取得（不可時は地図タップで指定）／地図・ハザード重畳表示
- **災害種別に応じた避難所ランキング**（推奨理由つき）
- **危険区域を避ける避難ルート**（洪水浸水域の回避）
- **現在地が浸水域内なら警告**（垂直避難の案内）
- **オフライン対応（PWA）**：一度保存した地域は圏外でも動作
- **OIDC 認証（Auth0）＋ HTTPS**

## 差別化の核 ①
### ハザードを考慮した避難所提案

---

## 避難所ランキングの中身

- その災害に **「指定された」避難所のみ** を候補にする
  （津波なのに非対応の低地施設を勧めない）
- **最寄り K 件に絞ってから評価** → 必ず近い順を担保
- **津波・高潮は標高で高台を優先**（国土地理院 標高API）
- 推奨理由を明示：`直線 約326m` / `地震の指定避難場所` / `標高 約25m（高台）`

<small>※ 距離は直線距離であることを明示し、徒歩時間は実ルートのみを正とする（過小表示を避ける）</small>

---

<!-- _class: section -->

## 差別化の核 ②
### 危険区域を避ける避難ルート

---

## 浸水を「実際に」避けるルート

- 洪水・台風時、**浸水想定区域(国土数値情報 A31)を避けて** 経路を引く
  （OpenRouteService の `avoid_polygons`）
- ルート周辺の浸水ポリゴンだけを送信し、API 制限内に収める
- **逃げ場が浸水で囲まれている場合** →
  「安全な経路が見つかりません。**垂直避難**を検討」と警告
- **現在地が浸水域内**なら最優先で警告（点 in ポリゴン判定）

> 「見えているだけ」でなく「実際に避ける」— ここが本アプリの核心

---

## 安全設計（ミスが許されない）

誤案内・過信を生む要素を **除去** する方針：

- **偽データを出さない**：架空のサンプルデータとフォールバックを完全排除
- **対象外を誘導しない**：内陸で津波を選ぶと「沿岸へ8km」ではなく垂直避難を案内
- **できないことを明示**：回避なしのルートは「自動回避していません」と表示
- **無反応を放置しない**：ルート探索にタイムアウト＋状況表示
- 「参考情報」であることと出典を常時明記

---

## システム構成（すべて無料・静的）

```
ブラウザ (PWA / Auth0でOIDC認証)
  ├─ 地図・ハザード：国土地理院タイル / 重ねるハザードマップ
  ├─ 避難所データ：地域ファイル(GeoJSON) → IndexedDBに保存
  ├─ 標高：国土地理院 標高API
  └─ ルート：OpenRouteService（洪水は avoid_polygons）
配信：GitHub Pages（静的・無料・HTTPS）
バックエンド／DB：なし
```

---

## オフライン設計 = 被災時の要件

**「平常時（オンライン）に備え、被災時（オフライン）に使う」**

- データを **都道府県ファイルに分割**、地点→地域は **bbox でローカル判定**（逆ジオコーディング不要）
- 検索した地域を **自動保存**、他地域は **手動で保存**（旅行前の備え）
- 保存は **IndexedDB**、アプリ本体・タイルは **Service Worker Cache**
- 一度保存すれば **圏外でも避難所検索が動く**

---

## 想定質問：なぜ DB を使わない？

- 災害時は **サーバ／通信が落ちうる** → サーバ DB は「必要な瞬間に頼れない」
- 読み取り専用の公開データなら、**静的分割＋端末キャッシュ**が最適・無料・無停止
- DB が要るのは **リアルタイム開設情報・ユーザー投稿・頻繁更新** を足すとき
  - その場合も **端末への配布は別途必要**（＝ハイブリッド）

> 本アプリの「オフライン要件」が、むしろ DB を不利にする

---

## 認証（必須要件）と HTTPS

- **OIDC（Auth0, Authorization Code + PKCE）** をバックエンドなしの静的サイトで実現
- 中核（地図・避難所検索）は **ログイン不要**
  （災害時にログイン画面が命の情報を塞がない設計）
- **HTTPS** は GitHub Pages で充足
- Google 連携（ソーシャルログイン）＝ OIDC フェデレーションも実演可能

---

## 使用データ・API（無料 / オープンデータ）

| 用途 | データ・API |
|---|---|
| 地図 | 国土地理院タイル |
| ハザード表示 | 重ねるハザードマップ（洪水/津波/高潮/土砂） |
| 避難所 | 国土地理院「指定緊急避難場所データ」(東京都 2,270件) |
| 洪水回避 | 国土数値情報「洪水浸水想定区域」(A31) |
| 標高 | 国土地理院 標高API |
| ルート探索 | OpenRouteService（無料枠） |

## 技術スタック

- **フロント**：HTML / CSS / Vanilla JS（ES Modules）、MapLibre GL JS
- **地図描画**：数千点は円レイヤ、上位候補のみピン（軽量）
- **オフライン**：Service Worker + Cache + IndexedDB（PWA）
- **認証**：Auth0 SPA SDK（OIDC / PKCE）
- **ホスティング**：GitHub Pages（静的・無料・HTTPS）
- **データ整形**：Node スクリプト（Shapefile/GeoJSON → 軽量化・簡略化）

---

## デモの流れ

1. ログイン（Google で続ける）※中核は未ログインでも可
2. 港区で「現在地から避難所をさがす」→ 近い順に理由つきで提案
3. **災害を切り替える**（地震 vs 洪水）とルート・優先度が変わる
4. **洪水**：東側の低地を起点にすると **浸水域を避けるルート**
5. **オフライン実演**：地域を保存 → 機内モードでも検索が動く

---

## 弱点・今後の展望

**正直に明示している限界**

- ルート回避は洪水のみ（土砂 A33 は今後）／「今その避難所が開設中か」は静的データでは不可
- 対応地域は東京都のみ（仕組みは全国対応、データを足すだけ）

**拡張**

- 他県データ追加／土砂災害の回避／地震ルート vs 洪水回避の比較モード
- 標高プロファイル／要配慮者モード（歩行速度・段差回避）

---

## まとめ

- **ハザードを考慮した避難意思決定支援**を、**完全無料・静的**で実装
- 差別化の核＝**危険区域を避けるルート**と**現在地の危険警告**
- 「ミスが許されない」前提の **安全設計**（誤案内・偽データの排除）
- **オフライン（PWA）× OIDC 認証 × HTTPS** を両立
- 設計思想「**平常時に備え、被災時に使う**」を体現

## 想定 Q&A

- **Q. Google マップと何が違う？**
  A. 危険（浸水・高台・指定避難場所）を考慮し、**回避ルート**と**推奨理由**を出す。
- **Q. 県境の避難所は？**
  A. 行政界分割の弱点。メッシュ／隣接地域バッファで補える設計。
- **Q. データ更新は？**
  A. 公開データから静的ファイルを再生成 → SW で更新。
- **Q. 認証があるのに静的？**
  A. OIDC はクライアント完結。データ配信と責務分離。
